# aion-context-studio — design context

## Design Context

### Users
Four operator roles share the studio, each in a different state of mind:
- **Policy authors / ops** — write and update the actual rules; want a fast, confident workbench.
- **Compliance / audit** — prove who-approved-what and review history; want a legible, trust-forward
  ledger and clean exportable reports.
- **Security engineers** — manage keys, the registry, multisig, integrity; want precise,
  instrument-grade controls and unambiguous state.
- **Developers integrating** — wire policies into agent loops; want API clarity and a
  simulate/test surface.
A `mode` switch tailors emphasis per role; the underlying data and design system are one.

### Brand Personality
Precise · verifiable · governed. This is a control surface for *signed policy* — the emotional
goal is **earned confidence**: every assertion (a signature, an approval, an integrity check) is
shown with its proof, never asserted. Calm, exact, institutional; hype is a liability. Honesty as
an asset — the local-demo / not-production-custody nature is stated plainly, as in aion-trust.

### Aesthetic Direction
**Light institutional — a "control ledger."** Family-consistent with aion-trust / aion-edu but a
distinct sibling: the palette derives from **aion-context's own brand** — deep navy ink
(`#0C141E`) and the logo's muted **teal** mark — not aion-trust's green.

- **Theme:** light (report-friendly for audit/compliance; readable for long policy text).
- **Palette (OKLCH, neutrals tinted toward the cool brand hue ~225):** cool off-white canvas; deep
  navy ink; a confident **teal** accent (the brand mark, deepened for light); semantics drawn for
  unambiguous state — verified/pass = teal-green, fail = oxblood, pending/warn = amber.
- **Type — distinct from aion-trust (no reuse of Libre Caslon / Hanken / Spline):**
  **Archivo** for headings (geometric grotesque, technical masthead character),
  **Public Sans** for body (humanist, the literal US-gov design-system face — apt for governance),
  and **JetBrains Mono** as a *load-bearing* mono for policy text, hashes, keys, DIDs, version ids.
- **Anti-references:** cyan-on-black "crypto dApp"; purple→blue gradients; neon/glow; gradient text;
  left-stripe alert bars; card-soup; monospace-as-decoration (here mono is functional, not vibe).

### Design Principles
1. **Proof over assertion.** The four verification guarantees, signatures, and approval thresholds
   are shown as evidence with their detail — verify-then-read, mirroring aion-context itself.
2. **State is unambiguous.** Valid / invalid / pending / revoked each have one unmistakable visual
   treatment; a security operator can read status at a glance.
3. **The policy text is the document.** Rules, diffs, hashes live in a precise mono with real
   hierarchy — the studio is a reading-and-editing instrument, not a dashboard of widgets.
4. **Role-aware, one system.** Each persona gets the right emphasis without forking the design.
5. **Honest about scope.** Local single-operator demo; keys server-side for the demo only; said
   plainly in UI and README. Claude advises and drafts — humans sign.
